What is usesuperpower

Not a plugin. A methodology.

Most AI coding tools make autocomplete smarter. usesuperpower gives your agent a complete engineering process — the same discipline senior engineers bring to production code, running automatically from first message to merged PR.

🧠

Thinks before it codes

Before writing a single line, it interviews you about what you're actually building. Proposes 3 approaches with trade-offs. Shows you the design in sections. Waits for your sign-off. You never get code for the wrong thing.

🤖

Executes autonomously

Once you approve the plan, it runs without stopping. A fresh subagent per task. A reviewer subagent after each. A fix subagent for anything that fails. It can work for hours without asking "should I continue?"

🧪

Tests first, always

The TDD Iron Law is enforced at the skill level: no production code without a failing test first. Write code before the test? Delete it. Start over. This isn't a suggestion — it cannot be rationalized away.

🔍

Reviews every task

After each task, a dedicated reviewer subagent — not the one that built it — checks spec compliance and code quality. Critical issues get fixed before the next task starts. Problems can't compound.

Problems it solves

The 4 ways AI coding breaks down

Every AI coding session fails for the same four reasons. usesuperpower is built specifically to prevent each one.

Problem 01

Agent built the wrong thing

You describe something loosely, the agent fills in assumptions, and two hours later you have working code for a feature you didn't want. By then switching costs are real and you rationalize the wrong thing into "good enough."

✓ Fixed by brainstorming — mandatory design gate before any code
Problem 02

Context pollution across tasks

As a session grows, earlier mistakes and wrong assumptions quietly become facts. Later tasks inherit bad context. The agent drifts from the plan in ways that are hard to trace or undo.

✓ Fixed by fresh subagent per task — zero inherited context
Problem 03

Code that was never tested

Tests get written after the implementation, if at all. They end up testing the shape of the code rather than its behavior. Refactors break everything because the tests were coupled to implementation details.

✓ Fixed by TDD Iron Law — failing test required before any code
Problem 04

Guessing at bugs instead of diagnosing

When something breaks, the agent tries random fixes. Each guess might mask the real issue. Quick patches hide root causes and create new failure modes in the next session.

✓ Fixed by systematic-debugging — root cause required before any fix

Why use usesuperpower

Built around how AI coding actually fails

Every design decision traces back to a real failure mode — not an engineering ideal.

1

You stay in control at every gate that matters

Design approval and merge decision are yours. Everything between those two moments is autonomous. usesuperpower never surprises you with a merge or skips a step you haven't seen. Your CLAUDE.md or AGENTS.md always wins over any skill — user instructions are highest priority, always.

2

Fresh context prevents the spiral

Every task starts with a clean agent that receives only what it needs for that task. The orchestrator coordinates. Subagents execute. Reviewers review. No agent knows more than its job requires — so accumulated session assumptions can't poison later work. No other skill system does this.

3

Hard gates are actually hard

The brainstorming gate is tagged HARD-GATE in the skill file — not "please try to do this first." The TDD Iron Law says delete code written before its test — not "add tests when you can." These aren't suggestions with rationalization escape hatches. They're enforced in the process.

4

Works across 11 agents — nothing to learn

Claude Code, Codex App, Codex CLI, Cursor, Gemini CLI, Copilot CLI, Factory Droid, OpenCode, Kimi Code, Antigravity, and Pi. Skills are plain Markdown — they work with any agent that accepts instruction files. No new interface to learn. It's just your agent, with a methodology.

5

You can build your own skills on top

The writing-skills skill teaches you to add your own engineering patterns to the system using the same TDD approach — write a pressure scenario, watch an agent fail, write the skill, verify it works. Your team's practices become part of the methodology.

Usage guide

How to use it, simply

There are no commands to memorize. usesuperpower activates automatically. Your job is to describe, review, and decide.

01
Start

Describe what you want to build — in plain language

Just talk naturally. No command prefix. No special format. usesuperpower detects that you're about to build something and activates brainstorming before anything else happens.

  • Be as vague or specific as you want — it will ask for what it needs
  • If your project is too large, it helps you decompose it into sub-projects first
  • It reads your existing files and docs before asking anything
⊘ No code is written at this stage
02
Design

Review the design and give your approval

usesuperpower presents the design in sections, scaled to complexity. It proposes 3 approaches with trade-offs and recommends one. Review, request changes, then approve.

  • Design doc saved to docs/usesuperpower/specs/
  • It self-reviews for contradictions before showing you
  • You can request as many revisions as you need
⊘ Hard gate — implementation cannot start without your sign-off
03
Execute

Say go — then step away

Once you approve, usesuperpower creates an isolated worktree, writes a detailed plan, and begins autonomous execution. A fresh subagent per task, a reviewer after each, a fix agent if anything fails. It runs until all tasks are done.

  • Plan saved to docs/usesuperpower/plans/
  • Each task: implement → test → review → fix if needed → commit
  • Only stops if it hits a genuine blocker it cannot resolve
⊘ This phase can run for hours without your input
04
Ship

Choose how to handle the finished branch

When all tasks complete and the final review passes, usesuperpower verifies the full test suite and presents four clean options:

  • Merge locally — pull it into your main branch right now
  • Open a PR — push the branch and create a pull request
  • Keep the branch — leave it, you'll handle it later
  • Discard — you explored something, you don't need the code
⊘ Your decision — usesuperpower never merges unilaterally
05
Debug

Report a bug — it diagnoses before it fixes

If something isn't working, just describe it. usesuperpower activates systematic-debugging and works through a 4-phase root cause process before proposing any fix. You won't get a random patch that masks the real issue.

  • Reproduce → localize → reduce → fix → guard against recurrence
  • Every fix gets a regression test before the issue is closed
  • If it can't reproduce the bug, it asks for more context — never guesses
